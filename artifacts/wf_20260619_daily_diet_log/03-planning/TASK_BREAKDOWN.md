## 2. 세부 작업

### T01. 제품·기술 결정 승인

- PRD의 MVP 범위와 제외 범위를 검토한다.
- 로컬 전용 저장, React/TypeScript/Vite, IndexedDB 사용을 승인한다.
- 가져오기 정책(기본은 병합, 별도 전체 교체)을 결정한다.

### T02. 프로젝트 기반

- 앱, TypeScript, 라우팅, 스타일 기반을 구성한다.
- 린트·포맷·단위 테스트·E2E 명령을 정의한다.
- 모바일 하단 내비게이션과 공통 레이아웃을 만든다.

### T03. 도메인과 검증

- `DailyLog`, `MealEntry`, `ExerciseEntry`, `UserSettings` 타입을 구현한다.
- 입력 및 가져오기용 Zod 스키마를 구현한다.
- 열량 합계, 날짜 범위, 통계 시계열 순수 함수를 작성한다.

### T04. 저장 계층

- `DailyLogRepository`와 `SettingsRepository` 계약을 정의한다.
- Dexie 스키마와 버전 1 마이그레이션을 구현한다.
- CRUD, 날짜 키 upsert, 범위 조회를 구현하고 통합 테스트한다.

### T05. 일일 기록

- 오늘 날짜 기본 선택과 날짜 변경을 구현한다.
- 체중, 식사 반복 항목, 운동 반복 항목, 물, 컨디션, 메모 입력을 구현한다.
- 저장 중·완료·실패 상태와 저장하지 않은 변경 경고를 구현한다.

### T06. 기록 탐색

- 단일 기록 삭제 확인과 빈 상태를 구현한다.

### T07. 통계

### T08. 설정과 데이터 이동

## 4. 주요 위험과 대응

- 날짜 경계 오류: 모든 저장 키를 로컬 날짜 유틸리티 한 곳에서 생성하고 경계 테스트를 둔다.
- 브라우저 데이터 삭제: JSON 백업/복원과 로컬 저장 안내를 MVP에 포함한다.
- 입력 과다로 기록 포기: 기본 화면은 핵심 필드만 열고 선택 항목은 접을 수 있게 한다.
- 차트의 오해 가능성: 결측 체중을 0으로 채우지 않고 데이터 부족 안내를 표시한다.
- 기능 확장에 따른 복잡도: 상세 영양소, 로그인, AI 기능은 MVP 수용 기준 달성 전 추가하지 않는다.
