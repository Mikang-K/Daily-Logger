## 1. 설계 원칙

1. 오늘 기록을 가장 빠르게 끝낸다. 첫 화면에서 날짜, 핵심 요약, 입력 섹션, 저장 동작이 한 흐름으로 이어져야 한다.
2. 건강 수치를 평가하거나 단정하는 표현을 피한다. 목표 초과는 `나쁨`이 아니라 `목표보다 180 kcal 많음`처럼 중립적으로 표시한다.
3. 데이터 저장 위치와 상태를 명확히 보인다. 설정 화면과 최초 빈 상태에 `이 기기의 브라우저에만 저장됩니다`를 표시한다.
4. 색상만으로 상태를 전달하지 않는다. 아이콘, 텍스트 레이블, 모양 또는 선 패턴을 함께 사용한다.
5. MVP 범위를 유지한다. 음식 검색, 영양소 분석, 사진, 소셜, 연속 기록 보상, AI 코칭은 노출하지 않는다.

## 2. 정보 구조와 내비게이션

### 2.1 전역 구조

```text
App shell
├─ Header: 화면 제목, 선택 날짜 또는 보조 설명
├─ Main
│  ├─ 오늘 / 날짜별 편집
│  ├─ 기록
│  ├─ 통계
│  └─ 설정
├─ 저장 상태 알림 영역
└─ Primary navigation
```

라우트 권장값:

- `/today`: 오늘 기록. 날짜 변경 시 `/log/YYYY-MM-DD`로 이동한다.
- `/log/:date`: 선택 날짜 기록 편집.
- `/history`: 기록 목록 및 날짜 탐색.
- `/insights`: 7일/30일 통계.
- `/settings`: 목표와 데이터 관리.

### 2.2 모바일 내비게이션

- 4개 항목: `오늘`, `기록`, `통계`, `설정`.
- 화면 하단에 고정하며 최소 높이 64px와 `env(safe-area-inset-bottom)`을 반영한다.
- 각 항목은 아이콘과 텍스트를 함께 표시하고 최소 터치 영역은 44×44px이다.
- 현재 항목은 채운 아이콘/배경과 `aria-current="page"`로 구분한다.
- 긴 폼에서 저장 버튼과 겹치지 않도록 본문 하단 여백을 내비게이션 높이 이상 확보한다.

### 2.3 데스크톱 내비게이션

- 1024px 이상에서는 좌측 고정 사이드 내비게이션(폭 224px)으로 전환한다.
- 콘텐츠 최대 폭은 편집 화면 760px, 기록/통계 화면 960px이다.
- 사이드바와 콘텐츠 사이 간격은 32px이며 하단 내비게이션은 숨긴다.

## 4. 화면별 상세 명세

## 4.1 오늘 / 날짜별 기록 편집

### 화면 순서

1. 헤더: `오늘의 기록` 또는 `2026년 6월 18일 기록`.
2. 날짜 이동: 이전 날짜 버튼, 날짜 선택 버튼, 다음 날짜 버튼. 미래 날짜는 선택 불가.
3. 일일 요약 카드: 현재 섭취 열량, 목표 대비 차이, 체중, 기록 상태.
4. 체중 섹션.
5. 식사 섹션.
6. 운동 섹션.
7. 물·컨디션 섹션.
8. 자유 메모.
9. 삭제 동작(저장된 과거/오늘 기록에만 표시).
10. 고정 저장 바.

### 일일 요약 카드

- `총 섭취 1,540 kcal`를 가장 크게 표시한다.
- 목표가 있으면 `목표 1,700 kcal · 160 kcal 남음`처럼 산술 차이를 표시한다.
- 목표가 없으면 `설정에서 일일 목표를 추가할 수 있어요` 링크를 표시한다.
- 체중이 없으면 `체중 미입력`, 기록이 저장되지 않았으면 `저장 전`을 텍스트로 표시한다.
- 섭취 합계는 식사 항목 변경 즉시 갱신한다.

### 체중

- 레이블 `체중`, 숫자 입력, 우측 단위 `kg`.
- `inputmode="decimal"`, step `0.1`, 허용 범위 20~500.
- 목표 체중이 있으면 입력 아래 `목표 62.0 kg`을 보조 텍스트로 제공한다.
- 체중은 선택 항목이므로 비어 있을 때 오류로 표시하지 않는다.

### 식사

- 섹션 헤더 우측에 `식사 추가` 버튼을 둔다.
- 항목 카드는 식사 구분 칩, 음식명, 열량, 선택 메모 요약, 편집/삭제 메뉴로 구성한다.
- 목록 상단 또는 하단에 `총 1,540 kcal`를 항상 표시한다.
- 빈 상태: `아직 등록한 식사가 없어요`와 `식사 추가` 버튼.
- 추가/편집은 모바일에서는 하단 시트, 768px 이상에서는 중앙 다이얼로그를 사용한다.
- 폼 순서: 식사 구분(아침/점심/저녁/간식), 음식명, 열량(kcal), 메모, 취소/추가 또는 수정.
- 음식명은 필수, 열량은 0~10,000의 정수로 검증한다.
- 삭제는 항목명과 함께 `이 식사 기록을 삭제할까요?` 확인 다이얼로그를 표시한다.

### 운동

- 식사와 동일한 추가/편집 패턴을 사용한다.
- 카드에는 운동명, 시간, 선택 소모 열량을 표시한다.
- 폼 순서: 운동명, 시간(분), 소모 열량(kcal, 선택), 취소/추가 또는 수정.
- 운동명은 필수, 시간은 0~1,440분, 소모 열량은 0 이상으로 검증한다.

### 물·컨디션

- 물: 숫자 입력과 `ml` 단위. 250ml 빠른 추가 버튼은 데이터 모델을 변경하지 않고 현재 값에 더하는 UI 보조 동작으로 허용한다.
- 컨디션: 1~5 단일 선택. 각 값에 `매우 나쁨`, `나쁨`, `보통`, `좋음`, `매우 좋음` 텍스트를 제공한다.
- 이모지를 사용할 경우 장식용으로 처리하고 실제 레이블은 텍스트/접근성 이름으로 제공한다.

### 메모

- 레이블 `오늘의 메모`, placeholder `식사나 몸 상태에 대해 자유롭게 적어보세요`.
- 일반 텍스트로만 표시하며 마크업을 렌더링하지 않는다.

### 저장 바

- 모바일: 하단 내비게이션 위에 고정하며 `저장` 전체 폭 버튼과 상태 텍스트를 포함한다.
- 데스크톱: 콘텐츠 하단 또는 우측 상단에 sticky 영역으로 둔다.
- 상태: `변경사항 있음`, `저장 중…`, `저장됨 · 오전 8:15`, `저장 실패`.
- `저장 중`에는 중복 제출을 막는다. 실패 시 입력을 유지하고 `다시 시도`를 제공한다.
- 저장할 변경이 없으면 버튼을 비활성화하고 상태 문구를 유지한다.

## 4.2 기록

### 화면 구성

1. 헤더 `기록`.
2. 월 선택: 이전/다음 월 버튼과 `2026년 6월` 버튼.
3. 월간 달력.
4. 선택 날짜 또는 최근 기록 목록.

### 달력

- 7열 달력이며 요일 헤더를 제공한다. 각 날짜 버튼은 최소 44px 높이이다.
- 기록이 있는 날짜는 작은 점과 `기록 있음` 접근성 이름을 제공한다.
- 오늘은 외곽선, 선택 날짜는 채운 배경으로 구분한다.
- 미래 날짜는 비활성화한다.
- 기록 점만으로 내용을 추정하게 하지 않고, 선택 시 아래 요약 카드에 상세를 표시한다.

### 기록 목록/요약 카드

- 날짜, 총 섭취 열량, 체중(있을 때), 식사/운동 개수를 표시한다.
- 카드를 누르면 해당 날짜 편집 화면으로 이동한다.
- 최근 기록은 날짜 내림차순이다.
- 빈 달 상태: `이 달에는 저장된 기록이 없어요`와 오늘 기록 이동 버튼.

## 4.3 통계

### 화면 구성

1. 헤더 `통계`와 `저장된 기록만 집계합니다` 보조 문구.
2. 범위 segmented control: `7일`, `30일`.
3. 요약 카드: 기록 일수, 최신 체중, 기간 평균 섭취 열량.
4. 체중 추이 차트.
5. 일별 섭취 열량 차트.

### 공통 차트 규칙

- 차트 높이: 모바일 220px, 768px 이상 280px.
- 좌우 스크롤 없이 표시한다. 30일 축 레이블은 적절히 생략하되 데이터 포인트는 유지한다.
- 차트 제목, 기간, 단위, 범례를 텍스트로 제공한다.
- hover뿐 아니라 키보드 포커스/탭으로 데이터 값을 확인할 수 있어야 한다.
- 차트 바로 아래에 `표로 보기` 토글을 제공해 날짜/값 테이블을 열 수 있게 한다.
- 데이터가 0~1개면 추세선을 과장하지 않고 값 카드와 `추이를 보려면 기록이 더 필요해요`를 표시한다.

### 체중 추이

- 녹색 실선과 원형 마커. y축 단위는 kg.
- 값이 없는 날짜는 0으로 연결하지 않고 선을 끊는다.
- 목표 체중이 설정되어 있으면 회색 점선과 `목표` 범례를 표시한다.
- 접근성 요약 예: `최근 7일 중 체중 기록 4일, 최신 67.2kg`.

### 섭취 열량

- 파란색 세로 막대. y축 단위는 kcal.
- 저장된 로그의 식사 합계를 사용한다. 로그가 없는 날짜와 식사가 0 kcal인 저장 로그를 시각적으로 구분한다.
- 일일 목표가 있으면 주황색 점선과 `목표 열량` 범례를 표시한다.
- 기간 평균은 저장된 기록 일수 기준이며 카드에 산정 기준을 명시한다.

## 4.4 설정

### 목표 설정

- `목표 체중(kg)`, `일일 목표 열량(kcal)` 입력과 `설정 저장` 버튼.
- 입력 범위와 단위는 항상 레이블에 포함한다.
- 저장 성공은 인라인 상태와 토스트로 표시한다.

### 데이터 관리

- 안내 카드: `기록은 이 브라우저에만 저장됩니다. 브라우저 데이터 삭제나 기기 변경 전에 내보내기를 권장합니다.`
- `JSON 내보내기`: 즉시 파일 생성. 파일명 권장값 `daily-diet-log-YYYY-MM-DD.json`.
- `JSON 가져오기`: 파일 선택 후 파일명, 로그 수, 기간을 검증 결과 화면에서 보여준다.
- 가져오기 방식은 `병합`을 기본 선택으로 하고 `기존 데이터 교체`는 별도 위험 선택으로 둔다.
- 교체 시 `기존 기록 N개가 교체됩니다` 확인과 체크박스를 요구한다.
- 검증 실패 시 기존 데이터가 변경되지 않았음을 명시한다.

### 위험 영역

- 페이지 마지막에 경계선으로 분리한 `위험 영역`을 둔다.
- `전체 데이터 삭제`는 danger outline 버튼으로 표시한다.
- 확인 다이얼로그에 삭제 범위와 복구 불가를 표시하고, `취소`에 기본 포커스를 둔다.

## 5. 반응형 동작

| 범위 | 레이아웃 |
| --- | --- |
| 360~479px | 단일 컬럼, 좌우 16px, 하단 내비게이션, 폼 전폭, 추가/편집 하단 시트 |
| 480~767px | 단일 컬럼 최대 640px, 요약 카드 일부 2열 허용 |
| 768~1023px | 콘텐츠 최대 760~960px, 요약 카드 3열, 추가/편집 다이얼로그 |
| 1024px 이상 | 224px 좌측 내비게이션, 콘텐츠 중앙 배치, 하단 내비게이션 숨김 |

- 필드가 나란히 배치되더라도 360px에서는 반드시 세로로 쌓인다.
- 가로 스크롤은 차트와 폼 어디에서도 기본 동작으로 사용하지 않는다.
- 키보드가 열린 모바일 환경에서도 고정 저장 바가 현재 입력을 가리지 않도록 `visualViewport` 또는 안전한 CSS 레이아웃을 적용한다.
- 200% 확대에서 콘텐츠와 기능이 손실되지 않아야 한다.

## 6. 상태와 피드백

| 상태 | 표현 | 동작 |
| --- | --- | --- |
| 초기 로딩 | 카드 스켈레톤 2~3개, `기록 불러오는 중` 비시각 텍스트 | 저장 동작 비활성화 |
| 빈 기록 | 설명, 핵심 CTA, 로컬 저장 안내 | 오늘 편집 또는 항목 추가 |
| 입력 오류 | 필드 경계, 오류 아이콘, 구체적 문구 | 첫 오류에 포커스, `aria-describedby` 연결 |
| 저장 중 | 버튼 spinner, `저장 중…` | 중복 제출 차단 |
| 저장 성공 | 인라인 `저장됨`과 짧은 토스트 | 포커스 강제 이동 없음 |
| 저장 실패 | 상단 오류 배너와 `다시 시도` | 입력값 유지, 오류 영역에 포커스 가능 |
| DB 초기화 실패 | 전체 페이지 오류 | `다시 시도`, 브라우저 저장소 안내 |
| 용량 부족 | 구체적 오류와 내보내기 안내 | 데이터 내보내기 링크 |
| 삭제/교체 | 확인 다이얼로그 | 취소 기본 포커스, 완료 후 결과 안내 |

- 페이지 이탈 시 저장하지 않은 변경사항이 있으면 앱 내부 이동과 브라우저 이탈 모두 확인한다.
- 로딩이 300ms 미만이면 스켈레톤을 지연 표시해 깜박임을 줄인다.
- 애니메이션은 150~200ms로 제한하고 `prefers-reduced-motion: reduce`에서 제거한다.

## 7. 접근성 구현 기준

- 문서 언어는 `ko`, 각 화면에는 하나의 `h1`, 섹션은 논리적 제목 순서를 유지한다.
- 모든 입력은 화면에 보이는 `<label>`과 연결한다. placeholder를 레이블로 사용하지 않는다.
- 오류 요약은 `role="alert"` 또는 적절한 live region으로 알리고 각 오류를 해당 필드와 연결한다.
- 저장 상태는 `aria-live="polite"`, 치명적 저장 오류는 `role="alert"`를 사용한다.
- 다이얼로그/하단 시트는 제목을 연결하고 포커스를 가두며 닫은 뒤 호출 버튼으로 복귀한다.
- 아이콘 단독 버튼에는 `aria-label`을 제공한다. 장식 아이콘은 `aria-hidden="true"`다.
- 날짜 선택, segmented control, 컨디션 선택은 가능한 한 네이티브 버튼/radio 패턴을 사용한다.
- 모든 상호작용 요소는 키보드로 도달하고 조작할 수 있으며 명확한 `:focus-visible` 스타일을 가진다.
- 터치 목표는 최소 44×44px, 인접 목표 간 간격은 최소 8px을 권장한다.
- 본문과 컨트롤은 WCAG 2.2 AA 대비를 충족한다.
- 차트에는 동등한 텍스트 요약과 표 데이터를 제공한다.
- 단위는 시각 텍스트와 접근성 이름에 모두 포함한다. 숫자만 읽히지 않게 한다.

## 8. 콘텐츠 문구 기준

- 저장 성공: `기록을 저장했어요.`
- 저장 실패: `기록을 저장하지 못했어요. 입력 내용은 유지됩니다.`
- 범위 오류: `20~500 사이의 체중을 입력해 주세요.`
- 식사 빈 상태: `아직 등록한 식사가 없어요.`
- 통계 빈 상태: `이 기간에는 저장된 기록이 없어요.`
- 가져오기 실패: `파일을 확인할 수 없어요. 기존 데이터는 변경되지 않았습니다.`
- 전체 삭제 확인: `모든 기록과 설정을 삭제할까요? 이 작업은 되돌릴 수 없습니다.`

문장은 짧고 행동 결과를 먼저 알린다. 체중·열량 수치에 도덕적 평가나 공포를 유발하는 문구를 붙이지 않는다.

## 9. 구현 완료 체크리스트

- 360px에서 모든 폼, 다이얼로그, 저장 버튼, 하단 내비게이션을 사용할 수 있다.
- 오늘 기록의 추가·편집·삭제가 같은 화면의 합계에 즉시 반영된다.
- 각 저장 상태와 실패 복구 동작이 구현되어 있다.
- 기록 달력에서 기록 유무, 오늘, 선택 날짜가 색상 외 방식으로 구분된다.
- 체중 결측치가 0kg으로 표시되지 않고 선이 끊긴다.
- 7일/30일 차트에 키보드 접근 가능한 값과 표 대체 표현이 있다.
- 가져오기 병합/교체와 전체 삭제에 명시적 확인이 있다.
- 폼 레이블, 오류 연결, 포커스 복귀, live region을 자동/수동 검사한다.
- 외부 네트워크 전송, AI 코칭, 사진, 상세 영양소 UI가 포함되지 않는다.
